# P!NG 2.3.7 — Revert Summary

## Problem

The plugin stopped working after today's session. The convolution block in `PluginProcessor` was refactored across commits 2.3.0–2.3.2 (all on 9 April 2026), and the plugin broke as a result.

---

## Root Cause

Six commits introduced in one session today all touched the convolution block:

| Version | What changed |
|---------|-------------|
| 2.3.0 | Convolution block refactor — initial pass |
| 2.3.1 | Further convolution refactor |
| 2.3.2 | Further convolution refactor |
| 2.3.3 | Added universal silence trim for file IRs |
| 2.3.4 | Added `isRestoringState` + `parameterChanged` guard |
| 2.3.5 | Added `audioEnginePrepared` + deferred IR load in `prepareToPlay` |
| 2.3.6 | Version bump only |

The 2.3.0–2.3.2 convolution refactor was the breakage. The 2.3.3–2.3.5 features were legitimate and needed to be kept.

## Fix Applied in 2.3.7

### Step 1 — Reverted the three source files to pre-2.3.0 (`7e4535e`)

```
Source/PluginProcessor.cpp
Source/PluginProcessor.h
Source/PluginEditor.cpp
```

Net effect: −70 lines of broken convolution refactor code, back to the working state.

### Step 2 — Re-applied the three features from 2.3.3–2.3.5

**Universal silence trim** (`loadIRFromBuffer`, from 2.3.3)  
Trims trailing silence from ALL IRs at load time (not just synth IRs). Critical for factory `.wav` IRs, which are synthesised at `8×RT60` (up to 60 s) but only contain 8–15 s of actual reverb. Without the trim, the NUPC background FFT thread cannot keep up at small buffer sizes (128–256 samples), causing persistent crackle on large-space presets. Threshold: −80 dB below peak, 200 ms safety tail, 300 ms floor.

**`isRestoringState` + `parameterChanged` guard** (from 2.3.4)  
`apvts.replaceState()` inside `setStateInformation` queues async `parameterChanged` notifications for every parameter, including `"stretch"` and `"decay"`. Without this guard, each fires `loadSelectedIR()`, producing 3 × 8 = 24 `loadImpulseResponse` calls in milliseconds — swamping the NUPC background thread and causing persistent crackling on any preset load. The flag is set at the start of `setStateInformation` and cleared via `MessageManager::callAsync` (after all queued notifications have fired).

**`audioEnginePrepared` + `deferConvolverLoad` + deferred callAsync in `prepareToPlay`** (from 2.3.5)  
Prevents a data race between `loadImpulseResponse` background threads and `reset()` on initial session open. On first load `setStateInformation` is called before `prepareToPlay`; calling `loadImpulseResponse` at that point spawns NUPC threads that race with `reset()`, corrupting internal convolver state and causing permanent distortion. Fix: `setStateInformation` saves state only (deferred path), then `prepareToPlay` sets `audioEnginePrepared = true` and posts a `callAsync` to fire the actual IR load after `reset()` completes.

### Step 3 — Version bumped to 2.3.7

`CMakeLists.txt` and `Installer/build_installer.sh` updated.

## What Was NOT Changed

- The `irLoadFadeBlocksRemaining` / `kIRLoadFadeBlocks` naming (block-based fade) was left as-is from the pre-2.3.0 code. The 2.3.x series had renamed these to `irLoadFadeSamplesRemaining` / `kIRLoadFadeSamples` (sample-based, which CLAUDE.md notes is more correct). This can be re-introduced as a standalone change in a future version.
- All other source files (`IRSynthEngine`, `PluginEditor`, UI components, tests) are unchanged.
